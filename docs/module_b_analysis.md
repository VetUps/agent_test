# Анализ задания и критериев для модуля Б

## Что именно трактуется как `2.1` и `2.2`

В конкурсном задании есть два разных блока с одинаковой нумерацией:

- раздел `2` в "Специальных правилах компетенции" — это про разрешённые и запрещённые предметы на площадке;
- раздел модуля Б — это `2.1 Построение аналитической системы` и `2.2 Разработка функционала аналитической системы`.

Для реализации агента использована именно вторая трактовка, потому что пользователь отдельно потребовал Dash-дашборд.

## Что требуют пункты 2.1 и 2.2

### `2.1 Построение аналитической системы`

Нужно сделать программный модуль, который:

- автоматически подключается к БД;
- строит интерактивный дашборд;
- регулярно обновляет данные;
- поддерживает уровни доступа;
- помогает быстро считывать ключевые метрики.

### `2.2 Разработка функционала аналитической системы`

Нужно показать шесть аналитик:

1. покрытие тем относительно среднего;
2. долю сгенерированных материалов по предметам и в целом;
3. распределение по типам занятий;
4. обеспеченность требований методических рекомендаций по категориям;
5. TOP наиболее и наименее выполняемых требований по предметам относительно среднего;
6. наиболее и наименее выполняемые требования для исходных и сгенерированных материалов.

## Что реально проверяют критерии

По листу оценивания для модуля Б выставляются баллы отдельно за:

- наличие интерактивного дашборда;
- чтение данных именно из БД;
- регулярное обновление;
- поддержку уровней доступа;
- наличие каждой из шести аналитик.

Это значит, что для проходного рабочего примера недостаточно просто нарисовать графики. Нужны:

- живое подключение к БД;
- автообновление;
- хотя бы базовая role-based access модель;
- понятная интерпретация методических категорий;
- заполненность тем, иначе покрытие будет выглядеть неполным и часть аналитики станет слабой.

## Особенности доступной БД `module_b`

В `module_b` доступны таблицы:

- `subjects`
- `topics`
- `materials`

На практике в исходных данных есть два ограничения:

- материалов меньше, чем тем;
- часть `materials.topic_id` не согласована с `subject_id`, поэтому прямой join по `topic_id` даёт неверные темы.

Из-за этого агент делает дополнительный шаг предобработки:

- проверяет, принадлежит ли `topic_id` нужному предмету;
- если нет, пытается восстановить тему по заголовку и аннотации материала;
- если покрытия всё равно не хватает, добавляет синтетические записи в отдельную таблицу `analytics_generated_materials`.

## Почему выбрана отдельная таблица для синтетики

База общая для нескольких модулей, поэтому базовую таблицу `materials` лучше не переписывать. Вместо этого агент:

- читает оригинальные материалы как есть;
- дозаписывает демонстрационные материалы в свою отдельную таблицу;
- строит дашборд поверх объединённого набора данных.

Такой подход безопаснее для конкурсной среды и нагляднее при демонстрации.
